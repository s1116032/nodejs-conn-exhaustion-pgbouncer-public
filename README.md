# IoT 高併發連線耗盡與 PgBouncer 擋煞實驗

## 📌 一、專案簡介
本專案旨在親手重現「連線耗盡」的災難現場，並透過 PgBouncer 實現「秒級重生」。
透過對比直連 PostgreSQL 與經過 PgBouncer 連線池的壓測結果，驗證連線池在高併發場景下的關鍵作用。

---

## 🎯 二、核心目標
1. 模擬 **200 個 IoT 設備** 同時上報資料。
2. 在 PostgreSQL 設定極低連線數 (`max_connections=20`) 的情況下，觀察直連帶來的災難性後果。
3. 透過 PgBouncer (`transaction` 模式) 攔截並管理連線，實現**零錯誤與穩定服務**。

---

## 🏗️ 三、實作架構與環境
- **操作系統**: Windows (Docker Desktop)
- **資料庫**: PostgreSQL 15-alpine (容器，`max_connections=20`)
- **連線池**: PgBouncer (容器，`pool_mode=transaction`, `default_pool_size=10`, `auth_type=scram-sha-256`)
- **測試工具**: Node.js + 自訂壓測腳本 (模擬 200 個併發設備)

---

## 🧪 四、測試場景設計
- **併發設備數**: 200 個
- **測試持續時間**: 30 秒
- **模擬操作**: 每個請求執行 `SELECT pg_sleep(1)` (模擬慢查詢，強制佔用連線 1 秒)
- **對比組別**:
  1. **對照組**: 壓測腳本直連 PostgreSQL (`localhost:5432`)
  2. **實驗組**: 壓測腳本連線 PgBouncer (`localhost:16432`)

---

## 📊 五、測試結果對比

### ❌ [對照組] 直連 PostgreSQL
| 評測項目 | 測試數據 |
| :--- | :--- |
| **測試持續時間** | 30 秒 |
| **成功請求數** | 0 |
| **失敗請求數** | 10,337 |
| **成功率** | **0.00%** |
| **吞吐量** | 0.00 請求/秒 |

#### 📝 結果分析
由於 200 個併發請求遠超 PostgreSQL 設定的 `max_connections (20)`，瞬間耗盡所有連線資源。PostgreSQL 回報 `"FATAL: remaining connection slots are reserved"` 錯誤，導致幾乎所有請求失敗。系統完全崩潰，無法提供任何有效服務。

---

### 🛡️ [實驗組] 經由 PgBouncer
| 評測項目 | 測試數據 |
| :--- | :--- |
| **測試持續時間** | 30 秒 |
| **成功請求數** | 487 |
| **失敗請求數** | 0 |
| **成功率** | **100.00%** |
| **吞吐量** | 16.23 請求/秒 |

#### 📝 結果分析
PgBouncer 成功將 200 個前端連線複用到後端有限的連線池 (10個) 上。請求在 PgBouncer 內部排隊，雖然因為 `pg_sleep(1)` 導致吞吐量受限 (約 16 req/s)，但**所有請求均成功執行，錯誤率為 0%**。資料庫受到完美保護，系統保持穩定運行，實現了「秒級重生」。

---

## 💡 六、關鍵結論
1. **連線耗盡是致命的**：當應用層併發數超過資料庫連線上限時，直連會導致服務完全不可用 (0% 成功率)，這在真實 IoT 場景中將造成業務中斷。
2. **PgBouncer 的擋煞效果**：PgBouncer 作為緩衝，將海量連線壓力吸收在自身，確保後端資料庫連線數始終在安全範圍內，是高併發架構的必備組件。
3. **穩定性 > 極致吞吐量**：雖然在慢查詢場景下 PgBouncer 的吞吐量看似不高 (16.23 req/s)，但這是在保證 100% 成功率的前提下。對照組雖然理論上能處理更多請求，但在耗盡連線後實際吞吐量為 0。

---

## 🛠️ 七、附錄: 環境配置注意事項
1. **Windows 端口衝突**：Windows Hyper-V 會保留某些端口範圍，`6432` 可能被佔用。解決方案是將 PgBouncer 容器映射到主機的 `16432` 端口。
2. **安全認證配置 (SCRAM-SHA-256)**：PostgreSQL 14+ 預設使用 `scram-sha-256` 密碼加密。為確保安全與相容性，PgBouncer 也配置為使用 `scram-sha-256` 認證類型 (`auth_type = scram-sha-256`)。PgBouncer 的 `userlist.txt` 中需使用明文密碼（僅供實驗）或正確的 SCRAM 密鑰。
